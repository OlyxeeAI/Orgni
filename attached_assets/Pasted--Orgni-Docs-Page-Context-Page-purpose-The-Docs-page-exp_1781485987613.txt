# Orgni Docs Page Context

## Page purpose

The Docs page explains how Orgni works, what developers can build with it, and how businesses can connect their information to the platform.

This page should not feel like generic developer documentation only. It should help both technical and non-technical users understand how Orgni turns business knowledge into structured intelligence.

## Main message

**Orgni helps businesses turn scattered information into structured context that workflows, finance tools, and AI systems can use.**

## What the Docs should cover

### 1. What Orgni is

Orgni is the core business-context platform.

It learns how an organization operates by reading documents, workflows, roles, rules, approvals, finance records, and operational explanations.

Orgni then converts that information into structured business intelligence that can be used by:

* Orgni Workflows
* Orgni Finance
* internal dashboards
* external AI systems
* business applications
* automation tools

### 2. Core concepts

The Docs should explain these concepts simply:

**Organization**
The business or company using Orgni.

**Sources**
Documents, spreadsheets, notes, voice explanations, connected systems, and uploaded files.

**Business Context**
The structured understanding of how the business works.

**Workflows**
Processes, approvals, handovers, tasks, escalations, and next actions.

**Finance Context**
Statements, ledgers, invoices, matches, exceptions, and review trails.

**Evidence Trail**
A record showing where each extracted fact came from.

**Confidence**
A score showing how reliable Orgni believes a piece of information is.

**Exceptions**
Items that need human review because Orgni cannot fully verify them.

### 3. How Orgni works

Use a simple flow:

Business inputs
→ Orgni reads and extracts context
→ Orgni builds a business map
→ Orgni identifies workflows and finance logic
→ Orgni creates structured outputs
→ Tools and AI systems use that context to execute safely

### 4. Getting started

The Docs should guide users through:

* creating an organization
* uploading business sources
* reviewing extracted context
* confirming roles, rules, and workflows
* connecting workflow or finance modules
* using the API to retrieve structured context

### 5. Use cases

Include short examples:

**Operations**
Upload process documents and Orgni identifies tasks, approvals, roles, and escalation paths.

**Finance**
Upload statements, ledgers, and invoices. Orgni helps identify matches, exceptions, and evidence.

**AI enablement**
Give AI systems reliable business context before they generate outputs or take action.

**Legacy businesses**
Help companies with old systems, spreadsheets, and manual processes become AI-ready.

## Tone

Clear. Technical when needed, but never complicated.

Avoid buzzwords. Explain every concept in practical business language.

# Orgni API Page Context

## Page purpose

The API page explains how developers can connect to Orgni and use its structured business context inside other systems.

The API should feel like infrastructure, not a chatbot API.

## Main message

**The Orgni API gives applications access to business context, workflows, finance intelligence, and source-backed evidence.**

## API positioning

Orgni API is for developers building systems that need to understand how a business operates.

It can be used to:

* create organizations
* upload business sources
* extract structured context
* retrieve workflow maps
* retrieve finance exceptions
* access source-backed evidence
* update business knowledge
* connect external tools or AI agents

## Main API sections

### 1. Authentication

Explain how developers authenticate requests.

Example topics:

* API keys
* organization-level access
* secure requests
* permission scopes

### 2. Organizations

Endpoints for creating and managing organizations.

Example:

* create organization
* get organization profile
* update organization details
* list departments and roles

### 3. Sources

Endpoints for uploading and managing business information.

Supported sources:

* PDF
* DOCX
* XLSX
* CSV
* images
* plain text
* voice transcripts
* connected system data

### 4. Context Extraction

Endpoints for turning raw sources into structured business context.

Outputs may include:

* roles
* departments
* rules
* policies
* workflows
* approval paths
* finance records
* exceptions
* missing information

### 5. Business Map

Endpoints for retrieving the organization’s operating map.

This should include:

* departments
* roles
* responsibilities
* workflows
* dependencies
* approval paths
* source references

### 6. Workflows

Endpoints for workflow-related context.

Examples:

* detected workflows
* workflow steps
* assigned roles
* approval rules
* escalation paths
* task states
* next actions

### 7. Finance

Endpoints for finance-related intelligence.

Examples:

* uploaded statements
* ledgers
* invoices
* reconciliation results
* matched records
* unmatched records
* exceptions
* audit trail

### 8. Evidence

Endpoints for checking where Orgni got its information.

Every important extracted fact should be traceable back to the source.

Example evidence fields:

* source document
* page number
* text snippet
* extraction confidence
* validation status
* timestamp

### 9. Updates

Endpoints for changing business knowledge over time.

Examples:

* replace old manager with new manager
* update approval rules
* change department structure
* correct extracted context
* mark information as verified

### 10. Webhooks

Orgni can notify external systems when important events happen.

Example events:

* source processed
* context extracted
* workflow detected
* finance exception created
* review required
* business context updated

## Example API flow

1. Create organization
2. Upload sources
3. Start context extraction
4. Review extracted business map
5. Fetch workflows
6. Fetch finance exceptions
7. Use evidence trail for verification
8. Push context into another system or AI tool

## Developer promise

**Build business-aware software without forcing the business to explain itself from scratch every time.**

## API tone

Precise, clean, and practical.

Avoid overexplaining AI. Focus on what developers can send, receive, verify, and build.
