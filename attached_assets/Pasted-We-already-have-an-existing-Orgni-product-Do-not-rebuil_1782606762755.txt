We already have an existing Orgni product. Do not rebuild it from scratch. Do not overwrite the current landing page, branding, layout, or working flows.

This task is incremental only.

Current product context:
Orgni already has:

* Existing marketing/landing page
* `/app` product area
* Express API backend
* PostgreSQL storage, with local lightweight store for dev
* Sources upload area
* Knowledge area
* Assistant named Lucy
* Profile/business setup popup
* Document parsing/chunking with provenance
* Deterministic mode and AI-powered mode
* Vercel deployment as one project

Important instruction:
Before changing anything, inspect the existing codebase and preserve what already works.

Do not:

* rewrite the landing page
* replace the UI system
* rename existing product concepts unless necessary
* remove working features
* break existing upload, parsing, assistant, or profile flows
* introduce a new architecture
* add marketplace/plugin complexity
* use dummy data as the main product flow

Goal:
Strengthen the existing Orgni app so it feels more like a reviewable operating model and less like only document Q&A.

Product direction:
Orgni turns a business’s documents, processes, and rules into a living operating model that can be reviewed, queried, and used to support workflows with evidence and control.

Work incrementally in this order:

1. Audit existing implementation first

Inspect:

* current routes/pages
* current database/schema/model layer
* current upload/source handling
* current Knowledge page
* current Assistant/Lucy behavior
* current business profile popup
* current deployment/build setup

Then make the smallest safe changes needed.

2. Preserve the landing page

Do not touch the public marketing/landing page unless there is a clear bug that blocks the app.

No redesigns.
No copy rewrites.
No branding changes.

3. Improve the existing Knowledge area

Do not replace it. Extend it.

Add review status to extracted findings if not already present:

* extracted
* review_needed
* approved
* rejected

Each finding should support:

* approve
* reject
* edit
* confidence score
* source document reference
* evidence text or chunk reference where available

If a full Knowledge Map schema already exists, extend it instead of creating a duplicate model.

4. Add or extend workflow support

If workflow objects already exist, extend them. Do not create a parallel system.

Minimum workflow support:

* view detected workflows
* create workflow manually
* edit workflow name and description
* edit steps
* save workflow
* reopen saved workflow
* export workflow as JSON
* status: draft, review_needed, approved

Only add what is missing.

5. Add basic exceptions without overbuilding

Add a simple Exceptions page or panel only if it fits the current app structure.

Exceptions should cover:

* low-confidence finding
* failed document parse
* missing required document
* conflicting rule
* approval gap
* unsupported assistant answer

Each exception should have:

* title
* type
* severity
* status
* related source/knowledge/workflow when available

Keep it simple.

6. Improve Home/Dashboard only inside the app

Do not touch the landing page.

Inside `/app`, add or improve summary cards:

* documents processed
* failed documents
* findings needing review
* approved findings
* workflows detected
* open exceptions
* Orgni confidence score if already supported or easy to derive

Avoid big redesigns. Use existing components/styles.

7. Keep Lucy grounded

Improve the existing assistant behavior incrementally.

Lucy must:

* answer only from uploaded documents/knowledge
* cite sources/pages/sections when available
* say when the answer is not found
* avoid inventing policies, roles, workflows, or rules

If no evidence supports an answer, either:

* respond that Orgni cannot confirm from current sources, or
* create/suggest an unsupported-answer exception if the exception system exists

8. Add lightweight audit history only if safe

If there is already activity/history support, extend it.

Track:

* knowledge approved/rejected/edited
* workflow created/edited/exported/approved
* exception opened/resolved
* source processed/failed

Do not block the MVP if this requires a large refactor.

9. Tests

Add focused tests around the new increments.

Required where practical:

* existing upload flow still works
* existing assistant still works
* existing profile popup still works
* knowledge item can be approved/rejected/edited
* workflow can be saved/reopened/exported
* exception can be opened/resolved
* assistant refuses unsupported answers
* business data remains scoped by businessId

10. Deployment stability

Keep the one-project Vercel deployment.

Before finishing, verify:

* local dev still runs
* production build passes
* existing landing page still loads
* `/app` still loads
* upload flow still works
* assistant still works

Deliverables:

* Summary of files changed
* What was added
* What existing behavior was preserved
* What still needs attention
* Any risky areas or assumptions

The existing Orgni product remains intact, but the app now has stronger reviewable knowledge, basic workflow management, basic exceptions, and stricter grounded assistant behavior.
